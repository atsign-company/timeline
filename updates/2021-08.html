rSDK (resilient Software Development Kit)

This week we are releasing a major update to our AtClient SDK (we call the rSDK) with corresponding updates to the SecondaryServer which has already been deployed. The primary goals of these updates are:

  • Improve the resilience and reporting of status conditions of operations.
  • Introduce the new NotificationService which takes the responsibility for availability and resilience of notification operations. It also provides improved status and error handling to help the application developers provide status feedback in their app.
  • Simplify our method signatures to make it easier for developers to use the SDK.
  • Improve interactive application performance with the new simplified method signatures and the addition of callback methods to avoid blocking for Dart “await” methods.
  • Improve the SyncService to more reliably synchronize large amounts of data and deal with connection anomalies.
  • Provide backward compatibility while adding the above improvements.

This coming week we will be working with the app developers to help them to implement and test these updates within their applications. If all goes well, we should finally be in a position with them to release reliable and performant applications to the stores that their customers can use and we can begin promoting in earnest. This will be valuable as we go forward for scale with more users on the platform and applications.

If you take a look at status.atsign.com you can see the infrastructure status for the @platform in real-time. This is not news to many of you, but what is news is that we hit 100% uptime for the last 90 days! 

@platform Resilience

The platform runs on infrastructure but also provides software infrastructure to the @signs using it. An individual reported that their secondary was acting up on Aug 16th, so we looked at it and fixed the issue, or so we thought. A second person reported the same issue on their @sign the next day and we knew we had a wider problem. After some work developing a tool to check every @sign automatically, we found a few more instances of the issue and the root cause (bug in the secondary code). 

By Aug 18th, the permanent fix was in place and we now have a bot to query all the @signs in our infrastructure every 15 minutes and report problems via Google Chat. This code will also be open sourced shortly for the people using dess to run on their own infrastructure. 

We always like to catch issues before they happen, but we also take any reports from people using the @platform, so if you see something please let us know.
