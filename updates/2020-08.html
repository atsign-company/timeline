Now for the @tech news:

@ Core Protocol

This month has been really exciting as we are joining the work streams together from different teams and seeing codebases interacting.

Some examples are compiling, testing, and automatically deploying the core protocol, the @me application using that core infrastructure, and the @me web interface displaying public information input into the @me app.

If that sounds like a lot of work and complexity, it is, but it also all works! Of course it is not without the odd bug. As we find those bugs and fix them the latest code automatically gets recompiled, tested and pushed to the infrastructure. This is all needed for automation and scaling of our platform to be ready for launch.  

All this works because of the core team’s discipline of managing branches of code, and that in turn allows multiple streams of work concurrently. This is also a measure of the maturity and the speed of development. In the early days we had just two branches of code. Now we have one per-sprint-branch and then development and production releases.

There has been no let up in pace on the creation of new features either, with the addition of caching of information shared with other @signs and critically, the revocation of that sharing and deletion of the shared information. All the new features exponentially enable new application possibilities in the future and are only limited by @ app developers creativity.

@ App Developers and the Software Development Kit

We started learning with the intern program how application developers would adopt the @ protocol and we learned a lot! We are continuing to learn as the members of our Early Access Program - young developers with huge ideas and no fear - start to work & create applications with the @ protocol. “The Tylers”, two of our former interns, are running the program and providing videos and code examples to help the EAP developers be productive and as fast as possible.

The biggest thing we have learned thus far is that while we meet developers who want to use the protocol directly, others just want to use the features of the @ protocol without having to know the gory details.

With that in mind this month we have developed a second higher level client API which abstracts the protocol to simple CRUDS operations. Create, Read, Update, Delete are simple well known operations but for good measure we added Scan.

Both the lower level API and the new higher level API work seamlessly; this new SDK further removes any perceived barriers of entry to using the @ protocol and speeds up the application developers adoption time and time to market for their MVP (minimal viable product).

As always, the months are flying past but we are making huge progress and appreciate everyone coming on this journey and adventure with us!
