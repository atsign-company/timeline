On to the techy stuff:

Getting free @signs within any @app are now possible using the latest release of the onboarding widget which uses a published API on the registrar website to allow people to select a free @sign quickly and simply (although we do ask for an email at present, we will not even ask for that in the near future). This has been integrated into the latest versions of the @buzz and @wavi (on the Google Play Store and via TestFlight on the App Store) apps. Please have a look and let us know what you think.

We have completed virtually all of the migration to the Dart null safetyrequirement for our published packages. The remaining ones are due to external dependencies and will be updated as soon as the dependent packages are released that comply with null safety.

Docker swarm mode is the technology we use to run and manage the thousands of microservices on the @platform. A couple of months ago we were contacted by the Docker team to say we were using it a little too much. We cycle around each microservice regularly to ensure it is running the latest code and each time we were doing that we pulled information from Dockerhub. Docker late last year changed the way they monetise and one new way is charging for pulls over certain limits.

This month, we implemented a caching proxy for Docker images that only pulls images if an update is required. This way the Docker team is happy and all our secondary server microservices are kept up to date automatically.

IoT ARM and SIM cards. One of our advisors connected us with a company in the Internet of Things market that has deep experience with Subscriber Identification Modules (SIMs). The idea of using SIM cards to store the @signs cryptographic keys is really interesting, as is using the ARM processor versions of dess. It is early, but we both see the potential for really interesting solutions where IoT devices become part of the “End-to-End Encrypted Everything” vision of the company.

TheStream verbis used to connect two @signs via a secondary and provide, what looks like to each device, an end to TCP socket connection. In actuality the secondary is relaying the IP packets between the two devices and can see the data being exchanged. To solve that problem we only encrypted data across the TCP socket. But what if the developer did not have to do that and the SDK took care of it? Enter the use of AES256 streaming mode and Dart’s converters and codecs. 
We can now ensure an encrypted stream from device to device, even if the developer does not encrypt the data before sending. While working on that code we also solved the problem of failed network connections forcing us to restart a whole download. As AES256 does not increase the number of bytes transferred, the stream verb asks for the starting point of the transfer, so if just the last two bytes got lost, they can be requested.

The stream verb updates are in testing and will go into production soon.

The Monitor verb has been made more robust and, like the stream verb, it can now reconnect and ask for any notifications it may have missed. This functionality is also in testing.
