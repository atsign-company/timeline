Now for the Techies!

For developers: 

Writing our code in Dart/Flutter has a number of huge advantages, including being able to run on pretty much any device and reuse code from the server side to the mobile device and back again seamlessly. But because both Dart and Flutter are also being developed as we develop our code, we have to keep up with the latest builds and ensure our code can take advantage of the latest versions. Null Safety is one such example.

Null Safety: Both Flutter and Dart have had some major changes in the last few months as Flutter hit the 2.0 milestone. Along with that milestone is something called “Null Safety”. Null Safety has the advantages of faster code and less chances of software failures, but we do have to rework all of our code to be null safe. Dart/Flutter developers have seen some of the larger packages becoming null safe and we are doing the same. It’s a bottom up task, so the lower level libraries need to be completed first and the widgets come at the end. It will take a while for us to complete everything but we are doing it so as not to be in debt!

@operations & @protocol core: 

Another form of technical debt is operational. To ensure that we keep everything clean operationally in our CI/CD we are fully utilizing GitHub.com. GitHub has a number of features that help us, including GitHub actions, which have been used to automate everything we can including building code, testing, and pushing to production. This can all be seen in our GitHub repos (for example the at_server repo).  The more we automate, the less technical debt we have as we grow.  

We have been investigating ways we can make the @protocol more reliable under less than perfect network conditions. We have focused on the notification and the monitor verbs and hope to roll out the new code very soon. Behind the scenes, the monitor verb code will take note of the last notification it saw and if the TCP connection drops, it can re-establish and know it has not missed anything. Similarly, the developer can know if the notification that they sent was actually received, along with retry strategies. The ultimate aim here is that the developer will not have to worry about network problems when the SDK and the @protocol are capable of handling them.
